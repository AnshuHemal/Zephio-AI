The Real Problem: Two Separate Issues
Issue 1 — Page Quality (Prompt Engineering)
Your WEB_GENERATION_PROMPT is already very strong — it has height rules, chart blueprints, variable mapping, and layout guidance. The quality problem is not the prompt structure, it's what the AI model is being asked to do in a single pass. Here's what's actually happening and how to fix it:

Root cause: The analysis step and generation step are disconnected

The WEB_ANALYSIS_PROMPT produces a visualDescription string, but the generation model receives it as plain text and has to interpret it. The description is often too abstract ("bento grid with glassmorphism") and the generator fills in the gaps with generic choices.

What to improve in WEB_ANALYSIS_PROMPT:

Force section-by-section breakdown. Right now visualDescription is a paragraph. Change it to require a structured list: navbar, hero, section_1, section_2, footer — each with exact Tailwind classes, exact content, exact icon names. The more specific the blueprint, the less the generator has to guess.

Add a contentPlan field to the JSON output. This should contain real copy — actual headlines, subheadlines, button text, feature names, testimonial names. Right now the generator invents content on the fly, which leads to generic "Lorem Ipsum"-style text even though you've banned it. Pre-generating the content in the analysis step and passing it to the generator eliminates this.

Add a componentLibrary field. List the exact components needed: ["pill-badge", "floating-navbar", "bento-grid-4col", "testimonial-carousel", "pricing-toggle", "footer-4col"]. This gives the generator a checklist to follow rather than improvising.

What to improve in WEB_GENERATION_PROMPT:

Add a "Section Completion Checklist" rule. Tell the model: every section must have a heading, supporting text, a visual element (icon, chart, image, or illustration), and a CTA or link. Sections with only text are not acceptable.

Add explicit "density" rules. The current prompt allows the model to generate sparse pages. Add: "Every section must fill its vertical space. No section should have fewer than 3 visual elements. Cards must have icons, labels, values, and trend indicators."

Add a "visual richness" requirement. Explicitly require: gradient overlays on hero sections, at least one SVG illustration or chart per page, hover states on all interactive elements, and a sticky/fixed navbar.

Strengthen the typography rules. Add specific size scales: H1 must be text-6xl or larger, section headings text-4xl, card titles text-xl. Right now the model often generates small, timid typography.

Issue 2 — Scroll Animations and Entry Animations
This is a completely separate problem from prompt quality, and it's the more impactful one for user perception. Here's the full picture:

Why there are no animations right now:

Your WEB_GENERATION_PROMPT explicitly says "Never use JavaScript". This is correct for safety (the iframe uses sandbox="allow-scripts" which does allow scripts, but you're being conservative). The problem is that CSS-only animations are limited — you can do entrance animations on load, but not scroll-triggered ones without JavaScript.

The solution: a self-contained animation script injected into every page

The right approach is to add a small, self-contained JavaScript snippet to your 
page-wrapper.ts
 (the getHTMLWrapper function). This script runs inside the iframe and handles all animations. You never ask the AI to write JavaScript — you inject it yourself, so it's always consistent and safe.

Here's what to inject:

1. Scroll-triggered entrance animations via IntersectionObserver

Add a <script> block to getHTMLWrapper that:

Selects all elements with a data-animate attribute (or a specific class like animate-on-scroll)
Uses IntersectionObserver to add an is-visible class when they enter the viewport
The CSS for is-visible transitions from opacity: 0; transform: translateY(24px) to opacity: 1; transform: translateY(0)
Then update WEB_GENERATION_PROMPT to tell the AI: "Add data-animate to every section, card, and feature block. Do not write any JavaScript — the animation system is already injected."

2. Staggered children animations

Add data-animate-stagger to grid containers. The injected script adds a transition-delay of 0ms, 80ms, 160ms, 240ms to each child, creating a cascade effect.

3. Smooth scroll behavior

Add scroll-behavior: smooth to the html element in getHTMLWrapper. This makes all anchor links (href="#") animate smoothly.

4. Parallax on hero sections

A 5-line scroll event listener that moves the hero background at 0.3x scroll speed creates a professional parallax feel with zero complexity.

5. Navbar scroll behavior

A scroll listener that adds a scrolled class to the navbar after 50px, which triggers a CSS transition to add backdrop-blur and reduce opacity — the standard "sticky nav" pattern.
